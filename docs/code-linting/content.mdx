## Features
### Consistent class order
In practice, you pay less attention to the order of classes, and as the number of classes increases, the template markup becomes less readable.

(i) The [class order rule](/docs/eslint#class-order) helps enforce a consistent and fixed order for classes.
<video className="aspect:2/1 r:5 max-w:xs" autoPlay loop muted playsInline>
    <source src="/videos/consistent-class-order.webm" type="video/webm" />
</video>

```mcss
font:12 font:24@sm m:32 block font:32@md mb:48
```
(o) Fix this @master/css/class-order problem.
```mcss
block font:12 font:24@sm font:32@md m:32 mb:48
```

### Syntax error checks
In the past, you would add utility classes to your templates in an unsafe manner, without being able to check whether they were valid or legal in the source code.

(i) The [class validation rule](/docs/eslint#class-order) helps you check the validity of classes early as you write them.
<video className="aspect:2/1 r:5 max-w:xs" autoPlay loop muted playsInline>
    <source src="/videos/syntax-error-checks.webm" type="video/webm" />
</video>

### Disallow traditional classes
You can also restrict teams to using [configuration](/docs/configuration) to create [styles](/docs/styles) instead of defining classes through traditional `html <style>` or CSS files.

(i) Based on syntax error checking, you can [disallow non-markup-driven traditional classes](/docs/eslint#class-validation).
<video className="aspect:2/1 r:5 max-w:xs" autoPlay loop muted playsInline>
    <source src="/videos/disallow-traditional-class.webm" type="video/webm" />
</video>

(x) <span className="text:error">Disallow a traditional class `card`.</span>
```html
<div class="btn">
```
(o) Use Master CSS syntax to fix this @master/css/class-validation problem.
```html
<div class="inline-flex">
```
(o) Or customize [styles](/docs/styles) to fix this problem.
```js name=master.css.js
export default {
    styles: {
+        btn: 'inline-flex'
    }
}
```
